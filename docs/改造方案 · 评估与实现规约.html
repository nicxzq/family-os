00  诊断与优先级

同一个病：承诺不清。两种药。

两个站都好看、有内容、有理念。缺的不是"做得更多"，而是让人一进门就知道：看什么、为什么看、看完能带走什么。

把朋友的话翻译成产品语言，两个站其实是两条不同的曲线：

站点现状下一步
wiki 个人知识库 自我整理：结构完整、主题清晰、有连续性 让读者快速进入 · 快速受益 · 快速复刻
family 家庭操作系统 温暖、正确、形式好，有传播潜力 让家人愿意一起用，且守住未成年隐私
最大的风险，先处理它

family 站涉及两个未成年孩子的年龄、习惯、家庭内部约定和具体场景。这是唯一一个"做错了无法撤回"的问题——内容一旦被抓取、被转发、被搜索引擎和 AI 收录，就收不回来了。所以隐私分区是 P0，排在所有"好看/好用"之前。它同时也正好是后面 AI 接口的访问边界——一举两得。

优先级总表
优先级事项站点为什么是这个级别
P0隐私分区：公开理念 / 家庭内部分层family涉及未成年，不可逆，必须最先做
P0去掉"满到做不到"的承诺（如"没有批评教育"）family低成本、高风险规避，和隐私一起做
P1首页价值承诺 + "从这里开始"wiki改动小、对"读者进入"提升最大
P1家庭共创模块《我们都能改》family把"爸爸的宣言"扭成"全家的系统"
P2《真实问题库》（公开方法 + 私密实例分层）family升维成"真实有用"，但要配合隐私模型
P2"AI 自动化部署"独立产品页wiki传播价值高，但工作量大，可稍后
P2AI 调用接口（llms.txt / JSON / MCP）两站建立在结构化 + 隐私分区之上，自然排后

A  意见逐条评估

哪些直接采纳，哪些要补一刀。

作为设计/产品视角，我对每条给一个明确判断，并说明理由——尤其是几条需要"在他基础上再进一步"的。

意见判断评估
wiki 首页加"价值承诺" 采纳 对。但他给的版本偏长、偏"简历感"。要再压一句，并把"你能带走什么"提到最前。见 B1。
wiki 加"第一次来从这里开始" 采纳 这是性价比最高的一处改动。建议做成 3 条路径（读者/复刻者/AI），不是一个按钮。见 B2。
wiki "AI 自动化部署"单独成页 采纳 同意，且它就是 wiki 当前最强的"可复刻"卖点。补充：把它和 AI 接口页打通，形成"看得懂→抄得走→调得动"的闭环。见 B3。
family 隐私边界要警惕 采纳并升级为 P0 这是全案最重要的一条。我把它从"注意"升级成"先做的架构决定"：用两层隐私模型替代零散删改。见 C1。
family 父亲视角太强，加"我们都能改" 采纳 对，且这条能直接治"爸爸的宣言"这个病。要做成真有改动记录的模块，而不是一句口号。见 C3。
family "没有批评教育"等过度承诺 采纳 完全同意。承诺要"留有余地、可被孩子反过来引用"。给出改写原则 + 逐句改写。见 C2。
family 加《真实问题库》 部分采纳 方向极好（从"温暖正确"到"真实有用"）。但它恰恰是隐私最敏感的内容：具体场景=家庭实例。所以必须拆成"公开方法层 + 私密实例层"，不能整块公开。见 C4。
整体：从"自我整理"转向"读者快速受益/复刻" 采纳 这是贯穿全案的北极星。AI 接口规划（D）正是"快速复刻/调用"的终极形态。
我唯一要补的一刀

朋友把《真实问题库》和《我们都能改》当成两个"加内容"的建议。我建议把它们和隐私模型绑成一个架构来做：先定"哪层公开、哪层私密"，再往里填内容。否则越往里写越真实、越真实越危险——会越做越被动。先有边界，再有内容。

B  wiki.carlxu.cn

让读者快速进入、受益、复刻。

三处改动，按性价比排序。每处都给"改什么"+"Codex 怎么落地"。沿用你 handoff.html 里已有的 token 与组件，不引入新体系。

B1 · 首页第一句：价值承诺

问题不是"没有理念"，而是理念在第三屏。第一屏要在 5 秒内回答"这是什么、对我有什么用"。朋友的版本是对的方向，但偏长。压成"一句主张 + 一句你能带走什么"：

现状 · 偏抽象
"一个普通人的第二大脑 / 个人知识库"——读者要读完才知道能干嘛。
改写 · 价值前置
我把十年写作、家庭教育、AI 工具和个人成长，整理成可复用的概念与流程。
你可以当它是一个普通人用 AI 搭第二大脑的公开样本——看得懂、抄得走、还能让 AI 直接调用。
两个可选语气（交给你定）
  • 样本型（推荐）：强调"公开样本/可复刻"——呼应卡帕西式自建系统，吸引同类人。
  • 受益型："这里的每个概念都能直接用在你的写作、育儿和 AI 工作流里。"——更直给，但少了点人格。
Codex · B1 实现
  • 改 content/pages/home 的 hero lede 段：一句主张 + 一句"你能带走什么"，总长 ≤ 60 字。
  • 沿用 handoff 的 --serif hero + --brick 关键词强调；同屏 brick 不超过 5 处（"可复用的概念与流程""公开样本"两处即可）。
  • 同步 <meta name="description"> 与 OG 描述为同一句主张，保证分享卡片一致。

B2 · "第一次来，从这里开始"

不要做成一个孤零零的按钮。读者进来其实分三种人，给三条清晰的路。放在 hero 正下方、第一屏内：

新增模块 · 放在 Hero 下方
第一次来？看你是哪种人 ——
1
想读点有用的 → 从 3 个代表作概念进
手挑 3 个最成熟、最能代表你的概念（如 富裕指数 / 从倦怠中恢复 / 时间当作朋友），直接给链接。读者 5 分钟就有收获。
2
想抄这套系统 → 去《如何用 AI 搭这个站》

3
你是 AI / 想让 AI 调用 → 看 llms.txt 与接口
通往 D 的 AI 接口。明确告诉机器读者"入口在哪"。
为什么是三条而不是一条

"降低门槛"的本质是替读者做分类。一个笼统的"开始"按钮，读者还是要猜。三条路各自只服务一种意图，反而更轻——这正是"读者一进来就知道该看什么"。

Codex · B2 实现

  • 3 个代表概念用 frontmatter 字段 featured: true 标记，首页查询 where featured == true 取前 3，避免硬编码。
  • 第 2、3 条分别链到 /about/ai-deploy（B3）与 /llms.txt（D）。

C  family.carlxu.cn

守住隐私，让全家愿意用。

顺序很重要：先立隐私边界（C1），再改承诺口径（C2），然后才往里加共创（C3）与真实问题库（C4）。

C1 · 隐私分区（P0，最先做）

不要用"这句删、那句改"的零散方式——会越改越漏。用一个两层模型一次定清楚，之后所有内容都先归层、再决定去哪。

Tier 0
公开理念层
任何陌生人看了都不会伤害到家人的东西
六条理念、方法论、共读书单、给"普通家庭"的通用做法、对外故事（已脱敏）。这层是站点的"传播面"，也是唯一允许被搜索引擎 / AI 收录的内容。
→ 可公开 · 可被 AI 索引 · 进 llms.txt
Tier 1
家庭内部层
只对家人有意义、且可能伤害到孩子的东西
孩子的真实年龄/姓名、个人习惯、家庭内部约定、真实冲突场景、家庭会议记录、照片。默认不公开：要么放在需要登录/口令的私密区，要么彻底留在本地（Obsidian / 家庭群），站上只放脱敏后的"模式"。
✗ 不公开 · 不被索引 · 永不进任何 AI 接口
具体到现有内容，怎么归层
现有内容层处理
六条理念、首页、家庭墙Tier 0保留公开。这是传播面。
"12 岁哥哥""6 岁弟弟"等具体年龄Tier 1对外改成"给青春期前后的孩子""给学龄前的孩子"，去掉精确年龄。
给哥哥的"五条约定"、家庭内部规则Tier 1移到私密区或改写成通用版；对外只留"我们家有几条约定"这种元信息。
真实冲突场景 / 习惯（问题库素材）Tier 1见 C4：公开"方法"，私密"实例"。
孩子的画、家庭会议记录、照片Tier 1私密区，或仅家人可见。
技术上至少先做这三件（零成本）

① 全站 robots.txt + 私密页 <meta name="robots" content="noindex,noai,noimageai">，挡住搜索引擎和爬虫；② 精确年龄→年龄段，姓名→称呼；③ Tier 1 内容若要上线，放在 /private 并加口令（Vercel 的 password protection 或一个简单的边缘函数校验）。先挡住"被动泄露"，再谈功能。

Codex · C1 实现
  • 给每条内容加 frontmatter 字段 privacy: public | family；构建时 public 进站点地图与 llms.txt，family 排除。
  • 新增 public/robots.txt（Allow 公开、Disallow /private）与全局 noai meta 注入私密模板。
  • /private/* 路由走口令校验（Vercel Password Protection 或 middleware 校验 cookie）。
  • 批量替换：精确年龄 → 年龄段；真实姓名 → 称呼（哥哥/弟弟/妈妈）。

C2 · 去掉"满到做不到"的承诺

朋友点名的那句在 for-eldest.html 和 miniapp/storydata.js 里。问题不是这句不好，而是它把大人架在了一个迟早会塌的台子上。改写原则：承诺要留余地，且让孩子有权反过来引用它。

现状 · 满到做不到
"二 · 你犯错可以，被指责不行。这家里没有'批评教育'。"
改写 · 诚实且赋权
"二 · 你犯错可以。我们尽量先一起解决问题，而不是先指责你。如果哪天我们没忍住、先怪了你——你可以提醒我们这条。"

这样改有三个好处：① 承认大人会失控，是诚实的；② 把"监督权"交给孩子，正好呼应 C3 的共创；③ 真的吵了架，这页还站得住——孩子不会觉得"说一套做一套"。

同类需要体检的表达
  • "永远 / 从不 / 没有 …" 句式 → 换成"尽量 / 我们努力 / 如果没做到就…"。
  • 所有"这家里没有 X"→"这家里我们在练习少用 X"。
  • 给孩子的承诺，结尾尽量带一句"你可以监督/提醒我们"。
Codex · C2 实现
  • 改 for-eldest.html 第 327 行与 miniapp/storydata.js 第 27 行同一句（两处要同步）。
  • 全仓 grep 绝对化词：永远 | 从不 | 没有.*教育 | 一定，逐条按上面原则软化。
  • 把"你可以提醒我们"做成一个可复用的句末样式（如 .kid-veto 斜体 coral 小字），强化"孩子有否决权"的视觉语言。

C3 · 家庭共创模块《我们每个人都可以改这套系统》

治"爸爸的宣言"这个病，靠的不是加一句"欢迎大家参与"，而是让站上真的有别人的笔迹。做成一个有角色 + 改动记录的模块：

新增板块 · 放在首页"家庭墙"之后
这套系统，谁都能改。
妈
妈妈可以改任意一条措辞
家庭墙的 Tweaks 已经支持改字——把"谁改的"也记下来。
兄
哥哥可以对任何一条投"反对票"
每条理念下留一个"我不同意，因为…"的入口，他的反对意见公开展示（脱敏）。
弟
弟弟可以画一张画代替一段字
用 image-slot 留位，孩子的画就是内容。
老
爷爷奶奶可以补一句家训
"长辈寄语"区，一人一句。

每月家庭会议更新一次
页脚显示"上次更新：家庭会议 2026-06"，证明它活着。
关键：要有"改动记录"

一个"修订日志"区——"5 月：哥哥反对'非必要不竞争'，他觉得有些事就是要争。保留他的意见。"——比任何"欢迎共创"的口号都有力。它证明这不是爸爸一个人的主张，而是一家人在持续协商的系统。这条改动记录本身要走隐私分区：公开"有哪些改动"，敏感细节留 Tier 1。

Codex · C3 实现
  • 新增板块 index.html#coauthor：5 个角色行 + 一个"修订日志"列表。
  • 数据放 family-changelog.js（{date, who, change, privacy}），privacy:public 才渲染。
  • 每条理念卡片底部加"反对意见"入口；老二用 image-slot.js 占位收画。
  • 页脚加"上次更新：家庭会议 YYYY-MM"，从 changelog 最新一条自动取。

C4 · 《我们家的真实问题库》—— 公开方法，私密实例

朋友的结构很好：场景 + 以前怎么处理 + 错在哪 + 今后怎么处理 + 一句话家庭约定。但"场景"就是家庭实例，是 Tier 1。所以做成双层卡片：公开的是可复用的"模式"，私密的是你家的"实例"。

一条示例（注意公开/私密的切分）
问题库 · 第 03 条 · 公开方法层
当孩子把事情搞砸了，大人第一反应是指责
典型场景（脱敏）孩子打翻 / 弄坏 / 忘记了某件事，现场有点狼狈。
以前怎么处理先来一句"你怎么这么不小心"，情绪上去了，事还没解决。
错在哪用指责代替解决，伤了关系，也没教会他下次怎么做。
今后怎么处理先一起把事收拾了，再平静地问"下次我们怎么做能更稳"。
一句话家庭约定"先解决事，再聊情绪。"

🔒 私密层（仅家人可见 / 仅本地）：具体是谁、哪天、原话、孩子当时的反应——这些留在 Tier 1，不上公开站。

这是全案最需要克制的地方

越真实越有用，但也越容易写出"指名道姓的脆弱时刻"。规则：公开层只能出现"任何家庭都可能发生"的脱敏版本；任何能定位到具体孩子、具体某次的细节，一律 Tier 1。宁可少一点真实，也不要让孩子十年后在搜索引擎里读到自己当年的糗事。

Codex · C4 实现
  • 新内容类型 problem：frontmatter {scene, before, wrong, after, pact, privacy}，public 字段与 family 字段分开存。
  • 卡片组件 ProblemCard：默认只渲染 public 五栏；private 区需登录后才注水（fetch /private/problem/[id]）。
  • 列表页 /problems，按主题分组；每条顶部标"脱敏公开"。
  • 构建校验：problem 的 public 文本里禁止出现姓名 / 年龄 token（CI 里 grep 黑名单，命中即构建失败）。

D  AI 调用接口规划

内容即数据，隐私分区即访问边界。

"让 AI 能调用"的前提，你已经做对了一半：wiki 内容已经是 concept/process/article 结构化的。剩下的是把结构暴露成机器可读的入口，并用 Tier 0/1 当访问控制——所以这一节必须排在隐私分区之后。

一句话原则

只有 Tier 0（公开理念层）进任何 AI 接口；Tier 1（家庭内部层）永远不进。AI 接口的访问边界 = 隐私分区的边界。这意味着隐私模型一旦做对，AI 接口几乎是"免费"叠加上去的——这也是为什么先做 C1 是划算的。

D1 · 三层接口，由浅入深

层级给谁形态工作量
① llms.txt任何 AI 助手 / 爬虫一个纯文本清单，列出站点结构与重要链接（新兴事实标准）极小
② JSON / RSS程序化读取每个 concept/process 一个 JSON；全站一个 index；RSS for 文章小（构建时生成）

D2 · ① llms.txt（先做这个）

放在站点根：wiki.carlxu.cn/llms.txt。让 AI 一眼看懂"这站有什么、入口在哪"。构建时自动生成，只列 privacy:public 的内容。

D3 · ② JSON 端点

构建时为每条公开内容生成静态 JSON（零运行成本，纯静态托管即可）。schema 直接复用你 handoff 里的 frontmatter：

// GET /api/concept/wealth-index.json
{
  "type": "concept",
  "slug": "wealth-index",
  "title": "富裕指数",
  "domain": "财富",
  "maturity": 4,
  "updated": "2026-04-07",
  "summary": "一句话定义…",
  "body_md": "## 定义\n…",
  "related": [{ "slug": "time-as-resource", "relation": "基础" }],
  "privacy": "public",
  "license": "CC BY-NC-SA 4.0"
}

端点清单：/api/index.json（全量轻量列表）、/api/concept/[slug].json、/api/process/[slug].json、/api/search.json（Pagefind 已有索引可复用）。family 站只暴露一个 /api/principles.json（六条理念，Tier 0），其余一律不暴露。

D4 · ③ MCP Server（让 Agent 主动调用）

当你想让 Claude / 别的 Agent 主动"去你的 wiki 里查"，就给它一个 MCP server。三个工具 + 资源足够：

// MCP tools（建议）
search_concepts(query)        → 命中的概念/流程列表（仅 public）
get_concept(slug)             → 单条全文 JSON
list_processes(domain?)       → 流程清单

// MCP resources
concept://{slug}              → 可被 Agent 直接引用的资源

// 硬约束（写进 server 的访问层）
assert(item.privacy === "public")   // Tier 1 永远取不到
实现路径（最省力）

MCP server 不必新写后端：让它读你已经生成的静态 JSON（D3）即可。一个轻量 Node MCP server，内部就是 fetch('/api/...') + 过滤 privacy==public。这样"内容只有一份真相（frontmatter）"，llms.txt、JSON、MCP 都是它的投影，不会各说各话。

D5 · 给"机器读者"的访问控制清单

面向开放关闭
搜索引擎 / 通用爬虫Tier 0 页面（robots Allow）/private、所有 family Tier 1（noindex + noai）
AI 索引（llms.txt）wiki 全部公开内容 + family 六条理念family 一切内部内容
程序化 JSONwiki public · family principlesfamily problems / changelog 细节 / 约定
MCP Agentwiki public（带 license 字段）family 全部（family 不挂 MCP）
Codex · D 实现
  • 构建后处理：遍历 content/*，privacy==public 者输出 /api/*.json 与拼接 /llms.txt、/llms-full.txt。
  • family 仅生成 /api/principles.json；CI 校验 family 产物里不含年龄/姓名 token。
  • MCP server 单独小仓 carlxu-wiki-mcp：三工具，数据源 = 线上静态 JSON，统一过滤 privacy。
  • 每个 JSON 带 license 字段（沿用 README 的 CC BY-NC-SA），让 AI 知道引用规则。

E  落地顺序

先堵风险，再提清晰，最后开放调用。

严格按这个顺序，因为每一步都是下一步的地基——尤其隐私分区，它同时是 family 的安全底座和 AI 接口的访问边界。

阶段做什么站点产出
P0C1 隐私分区 + C2 去过度承诺（robots/noai、年龄脱敏、改写"没有批评教育"）family风险被堵住，可放心继续
P1B1 价值承诺 + B2 从这里开始 · C3 家庭共创模块wiki family读者进得来，家人愿意用
P2B3 AI 部署产品页 · C4 真实问题库（双层）wiki family可复刻、真实有用
P2+D 全套 AI 接口（llms.txt → JSON → MCP）两站内容可被 AI 调用
一句话总结

wiki 要从"自我整理"走向"看得懂 · 抄得走 · 调得动"；family 要从"爸爸的家庭教育宣言"走向"一家人持续协商、且守得住孩子隐私的操作系统"。两条线最后在 AI 接口处汇合：把公开的那一层，开放给人和机器；把家人的那一层，永远留给家人。
