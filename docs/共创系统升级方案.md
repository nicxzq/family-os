## 一、现状诊断

### 已有基础

| 文件 | 作用 |
|---|---|
| `supabase/schema.sql` | auth + profiles + RLS 已配置 |
| `fo-utils.js` | `FO.getMember()` 从 localStorage 读当前成员 |
| `fo-supabase.js` | `FO_DB.getSession()` 获取 Supabase 会话 |
| `login.html` | 家庭成员选择器 + Supabase auth 覆层 |
| `family-changelog.js` | 静态 JS 数组，`FAMILY_CHANGELOG`，爸爸手写 |
| `index.html #coauthor` | 角色说明文字 + 硬渲染的修订日志 |

### 三个层次的缺口

**1 · 身份层**
`profiles.role` 只有 `parent / child / friend`，无法区分：
- 爸爸 vs 妈妈（同是 parent）
- 哥哥 vs 弟弟（同是 child）
- 爷爷 vs 奶奶（同是 friend）

页面感知不到「当前是谁」，自然无法按人显示不同的操作按钮。

**2 · 能力层**
`#coauthor` 角色行只是静态文字："妈妈可以改任意一条措辞"——但点开页面，妈妈找不到任何可以点击、可以写字的地方。声明和能力之间存在落差。

**3 · 记录层**
`FAMILY_CHANGELOG` 是硬编码 JS 文件。每次改动必须爸爸手改文件后 git push，其他家庭成员无法直接留下痕迹。没有 Supabase 表存储动态贡献。

## 二、目标与权限矩阵

| 成员 | family_role | role | 能做什么 | 贡献类型 |
|---|---|---|---|---|
| 爸爸 | 爸爸 | parent | 编辑六条原则措辞；发布修订日志 | `edit` |
| 妈妈 | 妈妈 | parent | 同上 | `edit` |
| 哥哥 | 哥哥 | child | 对任意原则写"我不同意"或补充想法 | `objection` |
| 弟弟 | 弟弟 | child | 在自己的展示区留一张画的占位描述 | `drawing` |
| 爷爷/奶奶 | 爷爷/奶奶 | friend | 写一句家训，显示在 #coauthor 区 | `motto` |

**设计原则**：
- `parent` 是内容发布者，可直接修改并使之生效
- `child / friend` 是内容贡献者，留下附加内容（意见、图、家训），不覆盖原文
- 所有贡献实时写入 Supabase，修订日志自动更新，无需 git push

## 三、技术方案

### 3.1 Supabase Schema 扩展

#### 扩展 `profiles` 表（新增 `family_role`）

```sql
alter table public.profiles
  add column if not exists family_role text
    check (family_role in ('爸爸','妈妈','哥哥','弟弟','爷爷','奶奶','其他'));
```

`role`（parent/child/friend）继续用于权限控制；`family_role` 用于 UI 个性化和日志显示。

#### 新增 `contributions` 表

```sql
create table public.contributions (
  id          uuid primary key default gen_random_uuid(),
  author_id   uuid not null references public.profiles(id) on delete cascade,
  type        text not null
                check (type in ('edit','objection','motto','drawing','note')),
  target_id   text,     -- 'direction'|'reading'|'learn-together'|'soft-skills'|'attention'|'time'
  field       text,     -- 'subtitle'|'summary'，type='edit' 时使用
  old_value   text,     -- 编辑前的内容（edit 类型时必填）
  new_value   text not null,  -- 新内容或贡献正文
  summary     text,     -- 人可读的一句话描述（自动生成或手写）
  privacy     text not null default 'family'
                check (privacy in ('public','family')),
  status      text not null default 'published'
                check (status in ('published','archived')),
  created_at  timestamptz default now()
);

-- RLS
alter table public.contributions enable row level security;

-- 任何已登录成员可以插入自己的贡献
create policy "contributions_insert_own" on public.contributions
  for insert with check (auth.uid() = author_id);

-- 任何已登录成员可以读取所有 published 贡献
create policy "contributions_select" on public.contributions
  for select using (status = 'published');

-- 作者可以 archive 自己的
create policy "contributions_archive_own" on public.contributions
  for update using (auth.uid() = author_id);

-- parent 可以 archive 任何贡献（管理权）
create policy "contributions_archive_parent" on public.contributions
  for update using (
    exists (select 1 from public.profiles where id = auth.uid() and role = 'parent')
  );
```

### 3.3 `fo-utils.js` 扩展

登录时，把 Supabase profile 同步到 localStorage：

```js
async syncFromSupabase() {
  if (typeof FO_DB === 'undefined') return;
  const session = await FO_DB.getSession();
  if (!session) return;
  const profile = await FO_DB.getProfile(session.user.id);
  if (profile) FO.setMember(profile);
}
```

这样 `FO.getMember()` 就能拿到 `family_role` 字段。

## 四、关键决策点（需要你确认）

### Q1 · parent 的修改是立即生效还是需要审核？

- **方案 A（推荐）**：parent 提交的 `edit` 立即生效，覆盖页面显示内容。child/friend 的贡献自动发布为附加内容（不覆盖原文），无需审核。
- **方案 B**：所有贡献进草稿池，parent 点「纳入」后才在页面显示。

推荐 A——家庭内部信任度高，流程越简单越能用起来。

---

### Q2 · 六条原则的「正式文本」存在哪里？

- **方案 A（推荐）**：静态 `api/principles.json` 是基线，parent 的最新 `edit` 贡献动态覆盖页面显示（JS 加载后合并）。断线时 fallback 到静态文本。数据库只存增量。
- **方案 B**：新建 `principles` Supabase 表，原则文本迁移到数据库，静态文件不再维护。

推荐 A——最小改动，静态文件还能支撑 AI 接口（`/api/principles.json`）不受影响。

---

### Q3 · 弟弟的「画」如何上传？

- **方案 A（推荐）**：先纯文字占位（弟弟用文字描述想画什么），图片由爸爸手动替换文件。`type:'drawing'` 的 `new_value` 存描述文字。
- **方案 B**：配置 Supabase Storage bucket，支持直接上传图片文件。

推荐 A——弟弟年纪小，先把参与门槛降到最低；以后有需要再升级。

---

### Q4 · 修订日志的公开范围？

- **方案 A（推荐）**：`privacy:'public'` 的贡献（爸爸手动标注）显示给所有人；`privacy:'family'` 的贡献只有登录家庭成员可见。与现有 `FAMILY_CHANGELOG` 逻辑一致。
- **方案 B**：所有贡献均对外公开（更透明，但涉及孩子的意见可能不适合公开）。

推荐 A——孩子的异见和妈妈的修改建议属于家庭内部，默认 `privacy:'family'`；需要公开时爸爸手动标记。

## 六、认证层：微信公众号 OAuth（路径 B）

家人主要用微信，选用**微信公众号网页授权**作为主登录方式，配合现有 Supabase 用户体系。

### 6.1 重要前提（订阅号限制）

| 能力 | 订阅号 | 服务号 |
|---|---|---|
| 网页授权 OAuth | ✅ 可用 | ✅ 可用 |
| `snsapi_base`（静默拿 OpenID）| ✅ | ✅ |
| `snsapi_userinfo`（拿昵称头像）| ✅ | ✅ |
| **仅微信内置浏览器有效** | ⚠️ 是 | ⚠️ 是 |
| 手机浏览器 / PC 浏览器 | ❌ 不走公众号 OAuth | ❌ 同 |

**结论**：家人必须通过微信聊天窗口点链接或扫二维码打开，才能触发微信登录。直接用手机浏览器输网址无效，需要 fallback 到 email/邀请码。

### 6.2 完整登录流程

```
家人在微信里点链接/扫码
        │
        ▼
 family.carlxu.cn (微信内置浏览器)
        │
        ├─ 检测到微信浏览器 + 未登录
        │
        ▼
 跳转微信 OAuth (snsapi_base，静默)
 https://open.weixin.qq.com/connect/oauth2/authorize
   ?appid=APPID
   &redirect_uri=https://family.carlxu.cn/api/wechat-callback
   &scope=snsapi_base
   &state=...
        │
        ▼ 微信自动重定向（无需用户操作）
        │
 /api/wechat-callback?code=XXX&state=YYY
 (Vercel Serverless Function)
        │
        ├─ 用 AppID + AppSecret 换 openid
        │
        ├─ 查 wechat_bindings 表
        │      │
        │      ├─ 找到 → 找到对应 user_id → 生成 Supabase session → 跳回首页
        │      │
        │      └─ 没找到（新用户）→ 跳转 /login?openid=xxx（选择家庭角色）
        │
        ▼
 用户选择角色（妈妈 / 哥哥 / 爷爷 ...）
        │
        ▼
 写入 wechat_bindings + profiles → 完成注册 → 跳回首页
```

- 填写：`family.carlxu.cn`（**不要带 `https://`，不要带路径**）
- 点「下载」→ 会得到一个 `MP_verify_XXXXXXXX.txt` 文件
- 把文件名和内容告诉我，我放到仓库根目录（Vercel 会自动在 `family.carlxu.cn/MP_verify_XXX.txt` 提供此文件）

|---|---|---|

### 6.4 技术实现（Codex 负责）

#### 新增文件

| 文件 | 说明 |
|---|---|
| `api/wechat-callback.js` | Vercel Serverless Function：code → openid → Supabase session |
| `MP_verify_XXXXXXXX.txt` | 微信域名验证文件（你提供文件名后填入）|

#### Supabase Schema 新增

```sql
-- OpenID 绑定表（openid → Supabase user）
create table public.wechat_bindings (
  openid      text primary key,
  user_id     uuid not null references public.profiles(id) on delete cascade,
  created_at  timestamptz default now()
);

alter table public.wechat_bindings enable row level security;
-- 只允许 service role 操作（Edge Function 持有 service_role key）
-- 不添加面向普通用户的 policy
```

#### `api/wechat-callback.js` 核心逻辑（伪代码）

```js
export default async function handler(req, res) {
  const { code, state } = req.query;

  // 1. 用 code 换 openid
  const tokenRes = await fetch(
    `https://api.weixin.qq.com/sns/oauth2/access_token` +
    `?appid=${APPID}&secret=${APPSECRET}&code=${code}&grant_type=authorization_code`
  );
  const { openid } = await tokenRes.json();

  // 2. 查 wechat_bindings
  const { data: binding } = await supabaseAdmin
    .from('wechat_bindings').select('user_id').eq('openid', openid).single();

  if (binding) {
    // 3a. 已绑定 → 生成 magic link → 重定向（自动完成登录）
    const { data } = await supabaseAdmin.auth.admin
      .generateLink({ type: 'magiclink', email: /* 该 user 的 email */ });
    return res.redirect(data.properties.action_link);
  } else {
    // 3b. 未绑定 → 跳到角色选择页，携带 openid（用 state 加密后传递）
    return res.redirect(`/login?wechat=${encodeURIComponent(openid)}`);
  }
}
```

#### `login.html` 改动

- 新增「微信一键登录」按钮（仅在微信浏览器内显示，通过 UA 检测）
- 接收 `?wechat=xxx` 参数时进入「绑定角色」流程（首次注册专用）
- 绑定完成后写 `wechat_bindings` + `profiles`

#### `fo-utils.js` 改动

### 6.5 fallback 策略

非微信浏览器（手机浏览器、PC、Safari）保留现有 **邮箱 + 邀请码**登录路径。两条路最终都写入同一个 `profiles` 记录，共享角色和贡献数据。

---

### 6.6 安全边界

- `WECHAT_APPSECRET` 和 `SUPABASE_SERVICE_ROLE_KEY` **只能在 Vercel 环境变量里**，绝不进代码仓库
- `wechat_bindings` 表无面向客户端的 RLS policy，只有 service_role 可操作
- `state` 参数用于防 CSRF，Edge Function 生成随机串写 cookie，回调时校验
