# 给 Codex 的落地手册 · 两站改造

> 操作手册 · How to drive Codex
> 配套文档：《改造方案 · 评估与实现规约》

方案文件是"做什么"。这份是"怎么让 Codex 去做"：每个阶段一段**可直接粘贴的指令**，外加一条贯穿始终的红线——**wiki 改"生成规则"，绝不改"生成出来的产物"。**

---

## 00 · 一条心法：单一事实源

**内容只有一份真相，其余都是它的投影。**

wiki 的内容来自你的内部知识库。页面、JSON、llms.txt 都只是它"算出来的结果"。Codex 要改的永远是**算法（模板/构建脚本/约定）**，不是**结果**。

| 这是什么 | 谁是真相源 | Codex 可以动吗 |
|---|---|---|
| wiki 概念/流程正文 | 你的内部知识库（Obsidian/MDX） | ❌ 否 · 改了会被下次构建冲掉 |
| wiki 页面 HTML 产物 | 构建脚本生成 | ❌ 否 · 是产物 |
| wiki 模板 / 组件 / token | 仓库源码 | ✅ 是 · 这才是改造的着力点 |
| wiki frontmatter 约定（featured/privacy） | 仓库 schema + 内容 | ✅ 是 · 加字段，内容侧再回填 |
| wiki 构建后处理（llms.txt/JSON） | 构建脚本 | ✅ 是 · 新增一步，不碰内容 |
| family 页面 HTML | 就是手写源码本身 | ✅ 是 · 直接改文件 |

> **两个仓库，两种打法**：**wiki** = 改"生成规则"，保留自动构建（改模板/脚本/约定）。**family** = 手写站，直接改文件。所以下面的指令对两个仓库分开发，**不要让 Codex 同时跨两个仓库做一件事**。

## 01 · 开工前：给 Codex 什么上下文

1. 把《改造方案 · 评估与实现规约》和本手册放进各自仓库的 `/docs` 目录（或仓库根）。wiki 仓也放一份方案，方便它对照 B 和 D 两部分。
2. 每个仓库各开一个分支：`feat/p0-privacy`、`feat/p1-entry` …… 一个阶段一个分支，别一锅烩。
3. **wiki 第一步永远是"侦察"**：先让 Codex 输出它对构建管线的理解，你确认无误，再让它动手。

### 给 Codex 的统一开场（两个仓库都贴一遍）

```
阅读 /docs/改造方案 · 评估与实现规约.md 与 /docs/给 Codex 的落地手册.md。
本仓库是【wiki / family（二选一）】。
铁律：
1. 只做我点名的那个阶段，不要顺手改别的。
2. wiki 仓：禁止编辑任何"由构建生成的内容/页面产物"；只改模板、组件、
   token、frontmatter 约定、构建脚本。改前先说明你判断哪些是产物、哪些是源码。
3. 任何涉及孩子年龄/姓名/家庭内部信息的改动，默认更克制、更保护。
4. 每个改动给出：动了哪些文件、为什么、如何回滚。
先不要写代码，先复述你的执行计划，等我确认。
```

## 02 · wiki 红线：能改什么 / 绝不能改什么

方案里 B（首页/入口/部署页）和 D（AI 接口）的每一项，都能用"改规则不改产物"的方式实现。

| 方案条目 | 正确做法（改规则） | 错误做法（改产物） |
|---|---|---|
| B1 价值承诺 | 改首页**模板**的 hero 文案区 + meta/OG 模板 | 直接改生成出来的 `index.html` |
| B2 从这里开始 | 新增**组件** + 用 `featured` frontmatter 查询取 3 条 | 把 3 个概念链接硬编码进页面 |
| B3 AI 部署页 | 新增**一个内容源文件** + 模板，让它跟着管线一起构建 | 手写一个脱离管线的静态 HTML 扔进产物目录 |
| D 接口 | 在**构建脚本末尾**加一步：遍历内容产 JSON / llms.txt | 手写一份 llms.txt 静态文件（内容一变就过期） |
| 隐私字段 | 给 schema 加 `privacy`，构建时按它过滤 | 在产物里逐页删 |

> **判断"是不是产物"的简单标准**：问 Codex 一句——*"如果我现在重新跑一次构建，这个文件会被覆盖吗？"* 会 → 它是产物，不准手改；不会 → 它是源码，可以改。把这句写进它的工作约定里。

## 03 · 分阶段指令（可粘贴）

按 P0 → P2+ 顺序，一段一段喂。每段都先"出计划等确认"，再实现。

### wiki · 侦察（动手前必做）

```
目标：让我确认你理解了构建管线，之后才动手。
请输出：
1. 这个站用什么生成？（Astro / Next / 自写脚本 / 其它）入口命令是什么。
2. 内容真相源在哪个目录？哪些目录是构建产物（会被覆盖）？
3. 首页 hero、导航、页脚分别由哪个模板/组件渲染？
4. frontmatter 现有哪些字段？加 featured、privacy 字段要改哪里？
5. 构建脚本在哪一步结束？要在末尾追加"生成 JSON/llms.txt"该插在哪。
只回答，不改代码。
```
→ 你核对无误后，再放行下面的实现指令。

### family · P0 隐私 + 去过度承诺（最先做）

```
分支 feat/p0-privacy。按方案 C1 + C2 实现，仅限本仓：
1. 全仓加 frontmatter/数据字段 privacy: public | family（family 站若是手写 HTML，
   就用注释或一个 data 文件标注哪些块是 family 私密）。
2. 新增 public/robots.txt：Allow 公开页，Disallow /private；
   私密模板注入 <meta name="robots" content="noindex,noai,noimageai">。
3. 文案脱敏：精确年龄→年龄段（"12 岁"→"青春期前后"，"6 岁"→"学龄前"）；
   真实姓名→称呼。给出改动清单让我逐条确认，不要自作主张删内容。
4. 改写过度承诺：for-eldest.html 第 327 行 与 miniapp/storydata.js 第 27 行
   同一句"这家里没有'批评教育'"，按方案 C2 的"诚实且赋权"版改写，两处同步。
5. grep 绝对化词（永远/从不/没有.*教育/一定），列出命中处，逐条建议软化，我确认后再改。
先出计划与改动清单，等我确认再写。
```

### wiki · P1 入口（侦察确认后）

```
分支 feat/p1-entry。按方案 B1 + B2，只改"生成规则"，不改产物：
1. B1：在首页模板的 hero 区改 lede 文案为方案给的版本（一句主张+一句你能带走什么，
   ≤60 字）；同步 description/OG 模板为同一句。
2. B2：新增组件 StartHere，3 行；其中 2 条代表概念用 frontmatter featured:true 查询
   取前 3，不要硬编码；另两行分别链到 /about/ai-deploy 与 /llms.txt（先占位可空）。
3. 不要触碰任何构建产物目录。改完跑一次构建，确认产物正常再提交。
```

### family · P1 家庭共创

```
分支 feat/p1-coauthor。按方案 C3：
1. 首页"家庭墙"之后新增 #coauthor 板块：5 个角色行 + 一个"修订日志"列表。
2. 修订日志数据放 family-changelog.js（{date, who, change, privacy}），
   只渲染 privacy:public 的条目。
3. 每条理念卡底部加"反对意见"入口；用 image-slot.js 给老二留画图位。
4. 页脚加"上次更新：家庭会议 YYYY-MM"，自动取 changelog 最新一条。
```

### wiki · P2 部署页 + D 接口

```
分支 feat/p2-deploy-and-api。
A) 按方案 B3 新增内容源 content/pages/ai-deploy + 模板，跟随现有管线构建，
   架构图复用 FlowChart 组件，代码块加复制按钮。
B) 按方案 D 在构建脚本末尾追加一步（不碰内容）：
   - 遍历内容，privacy==public 者输出 /api/index.json 与 /api/{type}/{slug}.json
   - 拼接 /llms.txt 与 /llms-full.txt（只含 public）
   - family 仓另行只产出 /api/principles.json
   每个 JSON 带 license 字段（CC BY-NC-SA）。
C) MCP server 放独立小仓，数据源=线上静态 JSON，统一过滤 privacy==public，先不动本仓。
先出计划，确认构建插点正确再实现。
```

> **family 的《真实问题库》（C4）要单独、最后做**。它最敏感。等 P0 隐私模型在 family 仓跑通后，再单开 `feat/problem-library`，且务必让 Codex 实现"公开五栏脱敏 / 私密实例分离"，并在 CI 里加"public 文本禁含年龄·姓名 token，命中即构建失败"的校验。这一条你要亲自审每一条内容。

## 04 · 验收：确认自动部署没坏

每个 wiki 分支合并前，跑这张单。核心就一句：**从知识库重新构建一次，改造仍在、内容照常更新。**

| 检查 | 通过标准 |
|---|---|
| 重新跑一次完整构建 | 无报错；B1/B2/B3 改动仍在产物里 |
| 往知识库加一条新概念再构建 | 新概念正常出现在站点 + `/api` + `/llms.txt` |
| git diff 看产物目录 | Codex **没有手改**任何产物文件 |
| family：访问任一私密页 | 有 noindex/noai；公开页不含精确年龄/姓名 |
| 抓 `/llms.txt` 与各 JSON | 只含 public；family 仅 principles 一个端点 |
| CI（problem 库上线后） | public 文本含姓名/年龄会让构建失败 |

> **一句话给你自己**：wiki 这边，凡是 Codex 想"直接改一个页面让它好看一点"，都先拦下来问："这会被构建覆盖吗？" family 这边则相反——它就是手写站，放心让 Codex 改文件，但**每一处涉及孩子的内容，你亲自过目。**
