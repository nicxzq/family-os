# 小程序 v2 需求文档

> 2026-07-20 · 来源：Carl 的 10 项需求。范围：`miniprogram/` 为主，网站仅在内容源被引用处附带调整。
> 配套：`spec.md`（技术规格）、`plan.md`（优先级与实施计划）。

## R1 · Logo 换成动图

把小程序内所有 logo 换成新提供的动图（GIF），按使用位置适配尺寸；canvas 生成的分享图/海报页脚也要带 logo（canvas 无法播 GIF，用 GIF 静帧生成 PNG）。

- 现状：首页 hero 用 `assets/logo.png`；`utils/share-card.js` 所有导出图页脚画 `assets/logo-mark.png`。
- **状态：待文件。** 用户将把 GIF 放入仓库后补做。

## R2 · 全页面可分享

所有页面都实现 `onShareAppMessage`（转发给朋友/群）与 `onShareTimeline`（朋友圈），动态页（idea/reader/story）分享路径带参数，对方点开直达该内容。

- 现状：仅 role、idea 两页有 `onShareAppMessage`。

## R3 · 首页两处优化

1. 家庭墙存图按钮去掉"（六条合一）"描述。
2. 共读清单：点某本书后，书的介绍卡直接展示在**所点书架行的下方**，不是整个书架最后。

## R4 · 家庭称呼统一

妻子 → 妈妈；哥哥 → 老大；弟弟 → 老二。

**边界（重要）**：只改"指代家庭成员"的 UI 称呼（角色名、书单归属、栏目标题、注释）。不改读本/绘本**正文叙述**里的词——如读本里弗兰克尔的"妻子"、绘本故事里"弟弟跑进来"、扉页"给小弟弟"，改了会破坏原文。生成文件（`data/stories.js`、`data/readers.js`）不手改。

## R5 · 读本/绘本书架化 + 翻本导航

- 老大读本、老二绘本的入口（role 页对应块）改成书架样式（参照首页共读清单书架）。
- reader / story 详情页底部增加"上一本 / 下一本"按钮，按目录顺序在可读内容间跳转。

## R6 · 游戏上线（5 类各首个）

52 个游戏按 5 类（问答 / 物理 / 化学 / 程序 / 多学科）各上线**首个**：g01-direction（问答）、g22-physics（物理）、g21-chemistry（化学）、g41-blocks（程序）、g44-history-detective（多学科）。其余全部展示预告卡。

- 约束：个人主体小程序**不能用 web-view**，5 个游戏须原生重写。
- 决策：**简化版原生移植**——保留核心玩法与内容，不追求像素级还原（问答类近似等价，重 canvas 的物理/化学/编程做简化关卡）。

## R7 · 家庭周会改版

1. 顶部周切换条布局挤压（"← 上一周 / 2026 年 · 第 N 周 / 下一周 →"挤在一行），重排。
2. 支持快捷分享（页内分享按钮）。
3. 支持"从上周复制"：把上周已填内容一键带入本周。

## R8 · 隐私政策与免责说明

"我的"页增加隐私政策和免责说明入口与内容（数据仅本机存储、不采集上传；内容基于李笑来家庭教育材料、仅家庭内部学习使用等）。同时满足小程序审核对隐私说明的要求。

## R9 · 内容后端规划

目标：**不发版（不提审）即可更新内容**。

- 决策：**微信云开发**路线（个人主体可用、免域名免备案），替代旧方案（`docs/miniapp/小程序实现方案.md` 的阿里云静态 JSON）。
- 本期交付：架构规划（见 spec.md §9），实现为后续独立阶段。

## R10 · 文档与实施方式

以上需求整理为本文档 + spec.md + plan.md，按难度与优先级分批交由低成本模型（subagent）实现，每批验证后进入下一批。
