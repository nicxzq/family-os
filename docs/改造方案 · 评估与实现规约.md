# 两个站点的下一步：从"自己清楚"到"读者带得走"

> 评估 · 改造方案 · Codex 规约 · AI 接口规划
> 站点：wiki.carlxu.cn / family.carlxu.cn
> 最高优先级：未成年隐私（P0）
> 交付物：规约文档，可直接交给 Codex 落地

朋友的判断是对的，而且很关键：现在两个站都是 **"你自己很清楚，读者要慢慢猜"**。这份文档把他的意见逐条评估，然后给出 **wiki 与 family 的具体改造方案**、可直接交给 Codex 的实现规约，以及一套 **面向 AI 的调用接口规划**——它和"隐私分区"是同一件事的两面。

---

## 00 · 诊断与优先级

**同一个病：承诺不清。两种药。**

两个站都好看、有内容、有理念。缺的不是"做得更多"，而是让人一进门就知道：**看什么、为什么看、看完能带走什么。**

| 站点 | 现状 | 下一步 |
|---|---|---|
| **wiki** 个人知识库 | 自我整理：结构完整、主题清晰、有连续性 | 让读者**快速进入 · 快速受益 · 快速复刻** |
| **family** 家庭操作系统 | 温暖、正确、形式好，有传播潜力 | 让家人**愿意一起用**，且**守住未成年隐私** |

### ⚠️ 最大的风险，先处理它

family 站涉及两个未成年孩子的年龄、习惯、家庭内部约定和具体场景。**这是唯一一个"做错了无法撤回"的问题**——内容一旦被抓取、被转发、被搜索引擎和 AI 收录，就收不回来了。所以隐私分区是 **P0**，排在所有"好看/好用"之前。它同时也正好是后面 AI 接口的访问边界——一举两得。

### 优先级总表

| 优先级 | 事项 | 站点 | 为什么是这个级别 |
|---|---|---|---|
| **P0** | 隐私分区：公开理念 / 家庭内部分层 | family | 涉及未成年，不可逆，必须最先做 |
| **P0** | 去掉"满到做不到"的承诺（如"没有批评教育"） | family | 低成本、高风险规避，和隐私一起做 |
| **P1** | 首页价值承诺 + "从这里开始" | wiki | 改动小、对"读者进入"提升最大 |
| **P1** | 家庭共创模块《我们都能改》 | family | 把"爸爸的宣言"扭成"全家的系统" |
| **P2** | 《真实问题库》（公开方法 + 私密实例分层） | family | 升维成"真实有用"，但要配合隐私模型 |
| **P2** | "AI 自动化部署"独立产品页 | wiki | 传播价值高，但工作量大，可稍后 |
| **P2** | AI 调用接口（llms.txt / JSON / MCP） | 两站 | 建立在结构化 + 隐私分区之上，自然排后 |

## A · 意见逐条评估

作为设计/产品视角，对每条给一个明确判断，并说明理由——尤其几条需要"在他基础上再进一步"的。

| 意见 | 判断 | 评估 |
|---|---|---|
| **wiki** 首页加"价值承诺" | ✅ 采纳 | 对。但他给的版本偏长、偏"简历感"。要再压一句，并把"你能带走什么"提到最前。见 B1。 |
| **wiki** 加"第一次来从这里开始" | ✅ 采纳 | 性价比最高的一处改动。建议做成 3 条路径（读者/复刻者/AI），不是一个按钮。见 B2。 |
| **wiki** "AI 自动化部署"单独成页 | ✅ 采纳 | 同意，且它是 wiki 当前最强的"可复刻"卖点。补充：和 AI 接口页打通，形成"看得懂→抄得走→调得动"闭环。见 B3。 |
| **family** 隐私边界要警惕 | ⭐ 采纳并升级为 P0 | 全案最重要的一条。从"注意"升级成"先做的架构决定"：用**两层隐私模型**替代零散删改。见 C1。 |
| **family** 父亲视角太强，加"我们都能改" | ✅ 采纳 | 对，且能直接治"爸爸的宣言"这个病。要做成**真有改动记录**的模块，而不是一句口号。见 C3。 |
| **family** "没有批评教育"等过度承诺 | ✅ 采纳 | 完全同意。承诺要"留有余地、可被孩子反过来引用"。给出改写原则 + 逐句改写。见 C2。 |
| **family** 加《真实问题库》 | 🟡 部分采纳 | 方向极好（从"温暖正确"到"真实有用"）。但**它恰恰是隐私最敏感的内容**：具体场景=家庭实例。必须拆成"公开方法层 + 私密实例层"。见 C4。 |
| 整体：从"自我整理"转向"读者快速受益/复刻" | ✅ 采纳 | 贯穿全案的北极星。AI 接口规划（D）正是"快速复刻/调用"的终极形态。 |

### 我唯一要补的一刀

朋友把《真实问题库》和《我们都能改》当成两个"加内容"的建议。我建议把它们和隐私模型**绑成一个架构**来做：先定"哪层公开、哪层私密"，再往里填内容。否则越往里写越真实、越真实越危险——会越做越被动。**先有边界，再有内容。**

## B · wiki.carlxu.cn 改造方案

三处改动，按性价比排序。每处都给"改什么"+"Codex 怎么落地"。沿用现有 `handoff.html` 的 token 与组件，不引入新体系。

### B1 · 首页第一句：价值承诺

问题不是"没有理念"，而是**理念在第三屏**。第一屏要在 5 秒内回答"这是什么、对我有什么用"。压成"一句主张 + 一句你能带走什么"：

- **现状（偏抽象）**："一个普通人的第二大脑 / 个人知识库"——读者要读完才知道能干嘛。
- **改写（价值前置）**：
  > 我把十年写作、家庭教育、AI 工具和个人成长，整理成**可复用的概念与流程**。
  > 你可以当它是**一个普通人用 AI 搭第二大脑的公开样本**——看得懂、抄得走、还能让 AI 直接调用。

两个可选语气（交给你定）：
- **样本型（推荐）**：强调"公开样本/可复刻"——呼应卡帕西式自建系统，吸引同类人。
- **受益型**："这里的每个概念都能直接用在你的写作、育儿和 AI 工作流里。"——更直给，但少了点人格。

**Codex · B1 实现**
- 改 `content/pages/home` 的 hero `lede` 段：一句主张 + 一句"你能带走什么"，总长 ≤ 60 字。
- 沿用 handoff 的 `--serif` hero + `--brick` 关键词强调；同屏 brick 不超过 5 处。
- 同步 `<meta name="description">` 与 OG 描述为同一句主张，保证分享卡片一致。

### B2 · "第一次来，从这里开始"

不要做成一个孤零零的按钮。读者分三种人，给三条清晰的路，放在 hero 正下方、第一屏内：

1. **想读点有用的 → 从 3 个代表作概念进**：手挑 3 个最成熟、最能代表你的概念，直接给链接。读者 5 分钟就有收获。
2. **想抄这套系统 → 去《如何用 AI 搭这个站》**：通往 B3 的 AI 自动化部署页。
3. **你是 AI / 想让 AI 调用 → 看 llms.txt 与接口**：通往 D 的 AI 接口。

> **为什么是三条而不是一条**："降低门槛"的本质是**替读者做分类**。一个笼统的"开始"按钮，读者还是要猜。三条路各自只服务一种意图，反而更轻。

**Codex · B2 实现**
- 新增组件 `components/StartHere.astro`：3 行 grid，复用 LibraryRow 的行式排版，hover 整行 `paper-warm`，不加卡片阴影。
- 3 个代表概念用 frontmatter `featured: true` 标记，首页查询取前 3，避免硬编码。
- 第 2、3 条分别链到 `/about/ai-deploy`（B3）与 `/llms.txt`（D）。

### B3 · "用 AI 自动化部署"独立产品页

wiki 当前最被低估的资产，也是最能传播的"可复刻"卖点。按朋友给的骨架，补两块让它成为**产品页**而非"关于"页：可复制模版 + AI 可调用入口。

| 板块 | 内容 | 形式 |
|---|---|---|
| ① 起因 | 为什么自建、为什么用 AI、解决了什么痛 | 一段叙事 + 一句结论 |
| ② 结构/架构图 | Obsidian → git push → 构建 → Vercel → 子域 | 横向流程图（复用 FlowChart） |
| ③ 自动化清单 | 每步谁触发、用什么工具、AI 在哪介入 | 带 ✓ 的清单表 |
| ④ 踩坑记录 | 真实失败 + 解法 | 问题/原因/解法 三列表 |
| ⑤ 可复制模版 | 仓库结构、frontmatter 模版、关键 prompt、tokens.css | 代码块 + 一键复制 + GitHub 链接 |
| ⑥ AI 可调用 | 本站内容如何被 AI 读取（接 D） | llms.txt / JSON 端点说明 |
| ⑦ 下一步 | 路线图：还想自动化什么 | 列表，标 `planned` |

**Codex · B3 实现**
- 新页面 `pages/about/ai-deploy.astro`，路由 `/about/ai-deploy`；内容用 `content/pages/ai-deploy.mdx`。
- 架构图复用 handoff §4.8 `FlowChart`。
- "可复制模版"区每个 `<pre>` 用 handoff §7 的复制按钮交互。
- 设为 B2 第 2 条与 home 的入口；footer "关于"分组里列出。

## C · family.carlxu.cn 改造方案

顺序很重要：**先立隐私边界（C1），再改承诺口径（C2），然后才往里加共创（C3）与真实问题库（C4）。**

### C1 · 隐私分区（P0，最先做）

不要用"这句删、那句改"的零散方式——会越改越漏。用一个**两层模型**一次定清楚。

**Tier 0 · 公开理念层** —— 任何陌生人看了都不会伤害到家人的东西
- 六条理念、方法论、共读书单、给"普通家庭"的通用做法、对外故事（已脱敏）。
- 这层是站点的"传播面"，也是唯一允许被搜索引擎 / AI 收录的内容。
- → 可公开 · 可被 AI 索引 · 进 llms.txt

**Tier 1 · 家庭内部层** —— 只对家人有意义、且可能伤害到孩子的东西
- 孩子的真实年龄/姓名、个人习惯、家庭内部约定、真实冲突场景、家庭会议记录、照片。
- **默认不公开**：要么放需要登录/口令的私密区，要么彻底留在本地（Obsidian / 家庭群），站上只放脱敏后的"模式"。
- ✗ 不公开 · 不被索引 · 永不进任何 AI 接口

**具体到现有内容，怎么归层**

| 现有内容 | 层 | 处理 |
|---|---|---|
| 六条理念、首页、家庭墙 | Tier 0 | 保留公开。这是传播面。 |
| "12 岁哥哥""6 岁弟弟"等具体年龄 | Tier 1 | 对外改成"给青春期前后的孩子""给学龄前的孩子"，去掉精确年龄。 |
| 给哥哥的"五条约定"、家庭内部规则 | Tier 1 | 移到私密区或改写成通用版；对外只留"我们家有几条约定"这种元信息。 |
| 真实冲突场景 / 习惯（问题库素材） | Tier 1 | 见 C4：公开"方法"，私密"实例"。 |
| 孩子的画、家庭会议记录、照片 | Tier 1 | 私密区，或仅家人可见。 |

> **技术上至少先做这三件（零成本）**：① 全站 `robots.txt` + 私密页 `<meta name="robots" content="noindex,noai,noimageai">`；② 精确年龄→年龄段，姓名→称呼；③ Tier 1 内容若要上线，放在 `/private` 并加口令。**先挡住"被动泄露"，再谈功能。**

**Codex · C1 实现**
- 给每条内容加 frontmatter 字段 `privacy: public | family`；构建时 public 进站点地图与 llms.txt，family 排除。
- 新增 `public/robots.txt`（Allow 公开、Disallow `/private`）与全局 `noai` meta 注入私密模板。
- `/private/*` 路由走口令校验（Vercel Password Protection 或 middleware 校验 cookie）。
- 批量替换：精确年龄 → 年龄段；真实姓名 → 称呼（哥哥/弟弟/妈妈）。

### C2 · 去掉"满到做不到"的承诺

朋友点名的那句在 `for-eldest.html` 和 `miniapp/storydata.js` 里。问题不是这句不好，而是它**把大人架在了一个迟早会塌的台子上**。改写原则：**承诺要留余地，且让孩子有权反过来引用它。**

- **现状（满到做不到）**："二 · 你犯错可以，被指责不行。这家里没有'批评教育'。"
- **改写（诚实且赋权）**："二 · 你犯错可以。我们尽量先一起解决问题，而不是先指责你。**如果哪天我们没忍住、先怪了你——你可以提醒我们这条。**"

三个好处：① 承认大人会失控，是诚实的；② 把"监督权"交给孩子，呼应 C3 共创；③ 真吵架了这页还站得住——孩子不会觉得"说一套做一套"。

**同类需要体检的表达**
- "永远 / 从不 / 没有 …" 句式 → 换成"尽量 / 我们努力 / 如果没做到就…"。
- "这家里没有 X" → "这家里我们在练习少用 X"。
- 给孩子的承诺，结尾尽量带一句"你可以监督/提醒我们"。

**Codex · C2 实现**
- 改 `for-eldest.html` 第 327 行与 `miniapp/storydata.js` 第 27 行同一句（两处同步）。
- 全仓 grep 绝对化词：`永远 | 从不 | 没有.*教育 | 一定`，逐条软化。
- 把"你可以提醒我们"做成可复用句末样式（如 `.kid-veto` 斜体 coral 小字）。

### C3 · 家庭共创模块《我们每个人都可以改这套系统》

治"爸爸的宣言"这个病，靠的不是加一句"欢迎参与"，而是**让站上真的有别人的笔迹**。做成有**角色 + 改动记录**的模块：

- **妈妈**可以改任意一条措辞（家庭墙 Tweaks 已支持——把"谁改的"也记下来）。
- **哥哥**可以对任何一条投"反对票"（每条理念下留"我不同意，因为…"入口，脱敏后公开展示）。
- **弟弟**可以画一张画代替一段字（用 image-slot 留位）。
- **爷爷奶奶**可以补一句家训（"长辈寄语"区，一人一句）。
- **每月家庭会议**更新一次（页脚显示"上次更新：家庭会议 2026-06"）。

> **关键：要有"改动记录"**。一个"修订日志"区——*"5 月：哥哥反对'非必要不竞争'，他觉得有些事就是要争。保留他的意见。"*——比任何"欢迎共创"口号都有力。它证明这不是爸爸一个人的主张。这条改动记录本身也走隐私分区：公开"有哪些改动"，敏感细节留 Tier 1。

**Codex · C3 实现**
- 新增板块 `index.html#coauthor`：5 个角色行 + "修订日志"列表。
- 数据放 `family-changelog.js`（`{date, who, change, privacy}`），`privacy:public` 才渲染。
- 每条理念卡底部加"反对意见"入口；老二用 `image-slot.js` 占位收画。
- 页脚加"上次更新：家庭会议 YYYY-MM"，从 changelog 最新一条自动取。

### C4 · 《我们家的真实问题库》—— 公开方法，私密实例

朋友的结构很好：**场景 + 以前怎么处理 + 错在哪 + 今后怎么处理 + 一句话家庭约定**。但"场景"就是家庭实例，是 Tier 1。所以做成**双层卡片**：公开的是可复用"模式"，私密的是你家的"实例"。

**一条示例（公开方法层）**

> **问题库 · 第 03 条**：当孩子把事情搞砸了，大人第一反应是指责
>
> - **典型场景（脱敏）**：孩子打翻 / 弄坏 / 忘记了某件事，现场有点狼狈。
> - **以前怎么处理**：先来一句"你怎么这么不小心"，情绪上去了，事还没解决。
> - **错在哪**：用指责代替解决，伤了关系，也没教会他下次怎么做。
> - **今后怎么处理**：先一起把事收拾了，再平静地问"下次我们怎么做能更稳"。
> - **一句话家庭约定**："先解决事，再聊情绪。"
>
> 🔒 私密层（仅家人可见 / 仅本地）：具体是谁、哪天、原话、孩子当时的反应——留在 Tier 1，不上公开站。

> **这是全案最需要克制的地方**：公开层只能出现"任何家庭都可能发生"的脱敏版本；任何能定位到具体孩子、具体某次的细节，一律 Tier 1。宁可少一点真实，也不要让孩子十年后在搜索引擎里读到自己当年的糗事。

**Codex · C4 实现**
- 新内容类型 `problem`：frontmatter `{scene, before, wrong, after, pact, privacy}`，public 字段与 family 字段分开存。
- 卡片组件 `ProblemCard`：默认只渲染 public 五栏；private 区需登录后才注水。
- 列表页 `/problems`，按主题分组；每条顶部标"脱敏公开"。
- 构建校验：problem 的 public 文本里禁止出现姓名/年龄 token（CI grep 黑名单，命中即构建失败）。

## D · AI 调用接口规划

**内容即数据，隐私分区即访问边界。**

"让 AI 能调用"的前提你已经做对了一半：wiki 内容已经是 `concept/process/article` 结构化的。剩下的是**把结构暴露成机器可读的入口**，并用 Tier 0/1 当访问控制——所以这一节**必须排在隐私分区之后**。

> **一句话原则**：只有 Tier 0（公开理念层）进任何 AI 接口；Tier 1（家庭内部层）永远不进。AI 接口的访问边界 = 隐私分区的边界。隐私模型一旦做对，AI 接口几乎是"免费"叠加上去的。

### D1 · 三层接口，由浅入深

| 层级 | 给谁 | 形态 | 工作量 |
|---|---|---|---|
| **① llms.txt** | 任何 AI 助手/爬虫 | 纯文本清单，列站点结构与重要链接（新兴事实标准） | 极小 |
| **② JSON/RSS** | 程序化读取 | 每个 concept/process 一个 JSON；全站一个 index；RSS for 文章 | 小（构建时生成） |
| **③ MCP Server** | Claude / 支持 MCP 的 Agent | 把"搜索概念、取概念、列流程"做成 AI 能主动调用的工具 | 中 |

### D2 · ① llms.txt（先做这个）

放在站点根：`wiki.carlxu.cn/llms.txt`。构建时自动生成，只列 `privacy:public` 的内容。


### D3 · ② JSON 端点

构建时为每条公开内容生成静态 JSON（零运行成本）。schema 直接复用 frontmatter：

```json
// GET /api/concept/wealth-index.json
{
  "type": "concept",
  "slug": "wealth-index",
  "title": "富裕指数",
  "domain": "财富",
  "maturity": 4,
  "updated": "2026-04-07",
  "summary": "一句话定义…",
  "body_md": "## 定义\n…",
  "related": [{ "slug": "time-as-resource", "relation": "基础" }],
  "privacy": "public",
  "license": "CC BY-NC-SA 4.0"
}
```

端点清单：`/api/index.json`、`/api/concept/[slug].json`、`/api/process/[slug].json`、`/api/search.json`（复用 Pagefind 索引）。**family 站只暴露一个 `/api/principles.json`（六条理念，Tier 0），其余一律不暴露。**

### D4 · ③ MCP Server（让 Agent 主动调用）

```
// MCP tools（建议）
search_concepts(query)        → 命中的概念/流程列表（仅 public）
get_concept(slug)             → 单条全文 JSON
list_processes(domain?)       → 流程清单

// MCP resources
concept://{slug}              → 可被 Agent 直接引用的资源

// 硬约束（写进 server 的访问层）
assert(item.privacy === "public")   // Tier 1 永远取不到
```

> **实现路径（最省力）**：MCP server 不必新写后端，让它**读你已生成的静态 JSON**（D3）即可。一个轻量 Node MCP server，内部就是 `fetch('/api/...')` + 过滤 `privacy==public`。这样"内容只有一份真相（frontmatter）"，llms.txt、JSON、MCP 都是它的投影。

### D5 · 给"机器读者"的访问控制清单

| 面向 | 开放 | 关闭 |
|---|---|---|
| 搜索引擎/通用爬虫 | Tier 0 页面（robots Allow） | `/private`、所有 family Tier 1（noindex + noai） |
| AI 索引（llms.txt） | wiki 全部公开内容 + family 六条理念 | family 一切内部内容 |
| 程序化 JSON | wiki public · family principles | family problems / changelog 细节 / 约定 |
| MCP Agent | wiki public（带 license 字段） | family 全部（family 不挂 MCP） |

**Codex · D 实现**
- 构建后处理：遍历 `content/*`，`privacy==public` 者输出 `/api/*.json` 与拼接 `/llms.txt`、`/llms-full.txt`。
- family 仅生成 `/api/principles.json`；CI 校验 family 产物里不含年龄/姓名 token。
- MCP server 单独小仓 `carlxu-wiki-mcp`：三工具，数据源 = 线上静态 JSON，统一过滤 `privacy`。
- 每个 JSON 带 `license` 字段（沿用 CC BY-NC-SA），让 AI 知道引用规则。

## E · 落地顺序（P0→P3）

严格按这个顺序，因为每一步都是下一步的地基——尤其隐私分区，它同时是 family 的安全底座和 AI 接口的访问边界。

| 阶段 | 做什么 | 站点 | 产出 |
|---|---|---|---|
| **P0** | C1 隐私分区 + C2 去过度承诺 | family | 风险被堵住，可放心继续 |
| **P1** | B1 价值承诺 + B2 从这里开始 · C3 家庭共创 | wiki + family | 读者进得来，家人愿意用 |
| **P2** | B3 AI 部署产品页 · C4 真实问题库（双层） | wiki + family | 可复刻、真实有用 |
| **P2+** | D 全套 AI 接口（llms.txt → JSON → MCP） | 两站 | 内容可被 AI 调用 |

> **一句话总结**：wiki 要从"自我整理"走向**"看得懂 · 抄得走 · 调得动"**；family 要从"爸爸的家庭教育宣言"走向**"一家人持续协商、且守得住孩子隐私的操作系统"**。两条线最后在 AI 接口处汇合：**把公开的那一层，开放给人和机器；把家人的那一层，永远留给家人。**
